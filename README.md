# Асинхронный сервис процессинга платежей

Микросервис для асинхронной обработки платежей с гарантированной доставкой событий (Outbox Pattern), идемпотентностью, повторными попытками (Retry) и Dead Letter Queue (DLQ).

## Стек технологий
- **Язык:** Python 3.11+
- **Веб-фреймворк:** FastAPI + Pydantic v2
- **База данных:** PostgreSQL + SQLAlchemy 2.0 (асинхронный)
- **Брокер сообщений:** RabbitMQ + FastStream
- **Миграции:** Alembic
- **Контейнеризация:** Docker + Docker Compose

## Запуск проекта

1. Убедитесь, что на вашем компьютере установлены **Docker** и **Docker Compose**.

7. В проекте есть файл .env.example, содержащий тестовые значения для файла .env


## Использование API
1. Аутентификация
Все эндпоинты защищены статическим API-ключом. Необходимо передавать заголовок:
   ```bash
   X-API-Key: secret-api-key-123
2. Создание платежа
Эндпоинт: 
   ```bash
   POST /api/v1/payments

Заголовки:

Idempotency-Key: уникальный ключ для защиты от дублей

X-API-Key: ваш ключ аутентификации

Пример тела запроса:

Пример тела запроса:
   ```json
   {
     "amount": 100.50,
     "currency": "USD",
     "description": "Тестовый платеж",
     "metadata": {"user_id": 123},
     "webhook_url": "https://httpbin.org/status/200"
   }
   ```

Ответ (202 Accepted): Возвращает payment_id, статус и дату создания.

3. Получение статуса платежа

Эндпоинт:
   ```bash
   GET /api/v1/payments/{payment_id}
   ```
Заголовки:
   ```bash
   X-API-Key: ваш ключ аутентификации.
   ```
Ответ (200 OK): Детальная информация о платеже, включая актуальный статус (pending, succeeded, failed).

4. Архитектура системы

Система построена на асинхронной обработке с гарантированной доставкой (Outbox):

API принимает запрос, проверяет идемпотентность, сохраняет платеж в БД со статусом pending и создает запись в таблице outbox.

Outbox-воркер (фоновый процесс в контейнере API) каждую секунду забирает события из таблицы outbox и публикует их в очередь RabbitMQ (payments.new), после чего меняет их статус на sent.

Consumer (отдельный контейнер) слушает очередь payments.new.

Эмулирует обработку платежа (задержка 2–5 секунд).

С вероятностью 90% присваивает статус succeeded, с 10% — failed.

Обновляет статус в базе данных.

Отправляет webhook-уведомление на указанный webhook_url.

Retry + DLQ: При ошибке отправки webhook Consumer делает 3 попытки с экспоненциальной задержкой. После 3-х неудачных попыток сообщение перемещается в Dead Letter Queue (payments.dlq).
